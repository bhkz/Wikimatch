# 1. CONTEXTE PRODUIT : LE PIVOT V2

## 1.1. Ancienne direction abandonnée

L’ancien prototype était centré sur une logique proche d’un dashboard temps réel :

* activité Wikipédia ;
* volume d’edits ;
* pages qui chauffent ;
* score de buzz ou drama ;
* globe ou visualisation de langues ;
* “guerres” ou “bagarres” d’édition ;
* cockpit live ;
* confrontation de matchs ;
* flux d’activité.

Cette direction est désormais abandonnée comme proposition de valeur principale.

Le problème était fondamental :

> Dire qu’un joueur a reçu plusieurs modifications Wikipédia n’est pas intéressant si ces modifications sont uniquement des corrections, des virgules ou du bruit technique.

Le produit doit désormais se concentrer sur ce qui raconte réellement quelque chose.

## 1.2. Nouvelle direction V2

WikiMatch devient un **média data expérimental autour de la Coupe du monde 2026**.

Promesse principale :

> **Le match se joue sur le terrain.
> Son histoire s’écrit sur Wikipédia.**

Promesse secondaire :

> **Un même match. Plusieurs Wikipédias. Parfois plusieurs récits.**

Le produit cherche à montrer comment un événement footballistique :

1. entre dans Wikipédia ;
2. apparaît dans différentes éditions linguistiques ;
3. est décrit de manière similaire ou différente selon les articles ;
4. devient éventuellement instable dans un article précis ;
5. se stabilise après un match ;
6. révèle parfois un joueur ou une équipe fortement documenté dans une édition linguistique avant d’apparaître ailleurs ;
7. produit un récit encyclopédique observable et vérifiable.

## 1.3. Positionnement final

WikiMatch ne doit plus être pensé comme :

```text
Un dashboard qui détecte automatiquement ce qui bouge.
```

Il doit devenir :

```text
Un média data qui publie uniquement les histoires vérifiables
que les traces Wikipédia permettent réellement de raconter.
```

# 2. PRINCIPES PRODUIT NON NÉGOCIABLES

Ces principes structurent toute l’architecture future : frontend, backend, base de données, ingestion, APIs, IA et modération.

## 2.1. Une modification est une trace, pas une histoire

Une modification Wikipédia observée est une **trace**.

Elle peut être :

```text
MINEURE
Correction de ponctuation, lien, mise en page, détail technique
ou modification sans changement narratif identifié.

SUBSTANTIELLE
Ajout, retrait ou reformulation d’un contenu compréhensible
lié à un match, un joueur, une équipe ou un événement suivi.

RELIÉE À UNE HISTOIRE PUBLIÉE
Trace utilisée comme preuve dans une story validée publiquement.
```

Une story ne doit jamais être créée automatiquement uniquement parce que :

* une page reçoit beaucoup d’edits ;
* plusieurs langues modifient le même sujet ;
* un seuil d’activité est dépassé ;
* une IA détecte un sujet potentiellement intéressant ;
* un score de buzz ou de drama augmente.

## 2.2. Plusieurs éditions linguistiques actives ne signifient pas divergence

Si plusieurs articles linguistiques ajoutent la même information, il peut simplement s’agir d’une mise à jour convergente normale.

Exemple :

```text
JA ajoute le résultat du match.
EN ajoute le résultat du match.
FR ajoute le résultat du match.
```

Cela ne démontre aucune tension.

Une divergence publique ne peut être formulée que si :

* les articles comparés portent réellement sur le même sujet ou épisode ;
* les passages sont comparables ;
* les différences sont lisibles et vérifiables ;
* la formulation publique reste prudente.

Formulation correcte :

```text
L’édition anglaise mentionne une altercation.
L’édition espagnole mentionne la sanction.
Aucune mention équivalente n’est détectée
dans l’édition française observée à cet instant.
```

Formulations interdites :

```text
Les Anglais accusent le joueur.
L’Espagne minimise l’incident.
La France l’ignore.
```

## 2.3. Une instabilité concerne un même article

Une instabilité éditoriale ne doit jamais être déduite du simple fait que plusieurs langues modifient un sujet.

Elle est observable lorsque, sur un **même article**, autour d’un passage comparable :

* un contenu est ajouté ;
* puis supprimé ;
* puis restauré ;
* ou reformulé plusieurs fois ;
* éventuellement accompagné ou non de nouvelles sources.

Exemple valable :

```text
Article EN du joueur fictif :
22:48 · passage ajouté
22:51 · passage retiré
22:56 · passage réintroduit
23:03 · formulation sourcée
```

Vocabulaire public recommandé :

```text
ARTICLE INSTABLE
INSTABILITÉ OBSERVABLE
PASSAGE AJOUTÉ, RETIRÉ PUIS RESTAURÉ
```

Éviter le vocabulaire :

```text
GUERRE
BAGARRE
DRAMA
```

tant qu’un protocole plus fort et défendable n’est pas défini.

## 2.4. Une édition linguistique n’est pas un pays

Une édition linguistique Wikipédia est un ensemble d’articles dans une langue.

Elle ne représente jamais automatiquement :

* un pays ;
* une population ;
* une opinion publique ;
* un groupe national ;
* la localisation de ses contributeurs.

Le produit doit employer :

```text
édition française
édition anglaise
édition japonaise
article comparé
```

Il ne doit jamais écrire :

```text
la France pense
les Anglais racontent
les Japonais réagissent
```

## 2.5. La carte situe les sujets, jamais les contributeurs

La page Explorer contient une carte.

Cette carte peut situer :

* une sélection concernée par une histoire ;
* un joueur associé à une sélection ;
* un stade ou lieu de match si cela est pertinent ;
* une entité footballistique clairement géolocalisable.

Elle ne doit jamais situer :

* la provenance d’un edit ;
* une IP ;
* le lieu supposé d’un contributeur ;
* une langue positionnée sur une zone géographique ;
* l’origine supposée d’un récit.

Règle permanente :

```text
POSITION = SUJET DE L’HISTOIRE
JAMAIS LOCALISATION DU CONTRIBUTEUR
```

## 2.6. La chronologie ne prouve pas une causalité

Si une modification apparaît après un but, un carton rouge ou une qualification, le produit peut dire :

```text
Cette modification est observée après l’événement du match.
```

Il ne peut pas automatiquement dire :

```text
Cette modification a été causée par l’événement du match.
```

Toute causalité forte exige des preuves complémentaires et une validation éditoriale.

## 2.7. L’IA peut assister, jamais publier seule

L’IA pourra éventuellement servir dans un futur Desk privé pour :

* traduire un extrait ;
* proposer qu’un passage concerne un événement ;
* suggérer une comparaison ;
* résumer une modification ;
* détecter un candidat à analyser.

Mais elle ne doit jamais automatiquement :

* publier une story publique ;
* affirmer une causalité ;
* qualifier une controverse ;
* attribuer une intention à un contributeur ;
* déduire une opinion nationale ;
* exposer une traduction non vérifiée comme vérité définitive.

* si les requêtes Wikimedia respectent une politique de User-Agent correcte.

Règle cible :

```text

```

Si le legacy ne garantit pas cette fiabilité, classe la collecte comme :

```text
BEST EFFORT À RENFORCER AVANT USAGE PUBLIC
```

# 9. RLS, SÉCURITÉ ET PRIVACY À ÉVALUER

Si Supabase est conservé, propose des politiques RLS adaptées.

Si une distinction technique d’acteurs est nécessaire pour analyser une instabilité, elle devra être minimisée, privée et justifiée ultérieurement. Ne l’implémente pas sans validation spécifique.
